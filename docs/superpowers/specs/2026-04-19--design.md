# 平台账号中心迁移设计

日期：2026-04-19

## 背景

当前 `multi-cli-studio` 的模型管理仅支持静态 API Provider 配置：

- `openaiCompatible`
- `claude`
- `gemini`

其数据结构位于 `src/lib/models.ts`，核心是 `baseUrl + apiKey + models`。界面入口是：

- `src/pages/ModelProvidersPage.tsx`
- `src/pages/ModelProviderEditorPage.tsx`
- `/settings/model-providers`

`cockpit-tools-main` 则已经实现了完整的平台账号中心，包含：

- Codex 账号中心、实例管理、Wakeup、配额、分组、本地访问、模型提供方管理、导入导出
- Gemini 账号中心、实例管理、注入、启动命令、导入导出
- Kiro 账号中心、实例管理、注入、导入导出、额度展示
- OAuth 登录与本地回调流程

本次需求不是新增几个登录按钮，而是将 Codex/Gemini/Kiro 的整套平台账号能力迁入当前项目，并按当前项目的信息架构与运行时模型完成适配。

## 目标

将 `/settings/model-providers` 从“静态 Provider 配置页”升级为“统一平台中心”，同时满足：

1. 保留现有 API Provider 配置能力，不影响 `ModelChatPage`。
2. 新增 Codex/Gemini/Kiro 三个平台的完整账号中心能力。
3. 支持 OAuth 登录、账号导入导出、切换、刷新、标签/筛选等基础能力。
4. 支持各平台外围能力：
   - Codex：实例、本地访问、Wakeup、配额、分组、会话管理、模型提供方管理
   - Gemini：实例、注入、启动命令
   - Kiro：实例、注入、额度/credits 展示
5. Kiro 第一阶段不接入 `ModelChatPage`，仅完成账号中心/实例/注入侧能力。

## 非目标

本阶段不做以下事项：

1. 不将 Kiro 接入 `ModelChatPage`。
2. 不重写 `ModelChatPage` 的选择模型协议，仍以现有 `ModelProviderServiceType` 为主。
3. 不迁入 `cockpit-tools-main` 中与本需求无关的平台（如 Cursor、Windsurf、Zed、Workbuddy 等）。
4. 不在第一阶段统一所有平台的聊天传输层。

## 设计原则

1. 当前项目继续区分两类能力：
   - API Providers：服务 `ModelChatPage`
   - Platform Accounts：服务 CLI/桌面平台账号、实例、注入、本地访问等
2. 账号数据不写入现有 `AppSettings`。
3. 平台账号数据、OAuth 状态、实例信息采用独立 Tauri 存储与命令接口。
4. 前端尽量复用 `cockpit-tools-main` 的成熟页面、类型、服务、复用 hook。
5. 后端避免继续把平台逻辑堆进单个 `src-tauri/src/main.rs`，新增模块承载迁入能力。

## 信息架构

### 统一入口

保留路由 `/settings/model-providers`，但页面重构为统一平台中心，顶层包含两个层级：

1. `API Providers`
2. `Platform Accounts`

### API Providers

保留现有能力：

- `openaiCompatible`
- `claude`
- `gemini`

包括：

- Provider 列表
- Provider 编辑
- 启用 provider
- 刷新模型列表

### Platform Accounts

新增三个平台子页：

1. `Codex`
2. `Gemini`
3. `Kiro`

每个平台子页采用独立账户中心视图，并在统一平台中心内部切换展示。

## 前端设计

### 路由与页面

新增平台中心相关页面：

- `src/pages/PlatformCenterPage.tsx`
- `src/pages/platformAccounts/CodexAccountsPage.tsx`
- `src/pages/platformAccounts/GeminiAccountsPage.tsx`
- `src/pages/platformAccounts/KiroAccountsPage.tsx`

保留并内嵌现有页面：

- `src/pages/ModelProvidersPage.tsx`
- `src/pages/ModelProviderEditorPage.tsx`

路由策略：

- `/settings/model-providers` 默认进入统一平台中心
- `/settings/model-providers/api`
- `/settings/model-providers/api/new`
- `/settings/model-providers/api/:serviceType/:providerId`
- `/settings/model-providers/accounts/codex`
- `/settings/model-providers/accounts/gemini`
- `/settings/model-providers/accounts/kiro`

旧路由保留重定向，避免已有链接失效。

### 复用策略

优先迁入并适配以下前端资产：

- `cockpit-tools-main/src/hooks/useProviderAccountsPage.ts`
- `cockpit-tools-main/src/services/codexService.ts`
- `cockpit-tools-main/src/services/geminiService.ts`
- `cockpit-tools-main/src/services/kiroService.ts`
- `cockpit-tools-main/src/types/codex.ts`
- `cockpit-tools-main/src/types/gemini.ts`
- `cockpit-tools-main/src/types/kiro.ts`

按需迁入平台相关组件、工具和展示层：

- 标签编辑、导出弹窗、快速设置
- 概览 tabs
- 实例页/实例内容
- Codex 的 local access、wakeup、session manager、provider manager

### 状态管理

新增账号中心 store 域，不复用 `AppSettings`：

- `useCodexAccountStore`
- `useGeminiAccountStore`
- `useKiroAccountStore`

store 职责：

- 拉取账号列表
- 保存当前账号 ID
- 删除/批量删除
- 刷新单个/全部 token
- 更新 tags
- 与平台页事件同步

### 与当前项目的适配

1. `DesktopSettingsPage` 的 “模型管理” 入口改为统一平台中心。
2. `Sidebar` 与 `SettingsPage` 中所有跳转保留，但指向新入口。
3. `ModelChatPage` 仍只消费 API Provider；当用户需要管理聊天来源时，跳转到 `API Providers` 子页，而非平台账户子页。
4. 平台账户页与现有设置样式统一，不引入第二套顶级外壳。

## 后端设计

### 模块化迁入

在 `src-tauri/src/` 下拆分新模块，而不是把逻辑继续内联到 `main.rs`：

- `modules/platform_accounts/`
- `modules/platform_accounts/codex_*`
- `modules/platform_accounts/gemini_*`
- `modules/platform_accounts/kiro_*`
- `modules/platform_accounts/oauth_pending_state.rs`

### 命令范围

第一阶段接入以下命令族：

#### Codex

- 账号列表/当前账号/切换/删除/批量删除
- OAuth start / complete / cancel / submit callback
- token/API key 导入
- 导入导出
- 刷新 token / quota / profile
- tags
- 本地访问
- 实例管理
- Wakeup
- 会话管理
- 模型提供方管理

#### Gemini

- 账号列表/当前账号/切换/删除/批量删除
- OAuth start / complete / cancel / submit callback
- token 导入
- 导入导出
- 刷新 token
- tags
- 实例管理
- 注入与启动命令

#### Kiro

- 账号列表/当前账号/切换/删除/批量删除
- OAuth start / complete / cancel / submit callback
- token 导入
- 导入导出
- 刷新 token
- tags
- 实例管理
- 注入

### 存储策略

平台账号数据采用独立本地索引，不写入 `AppSettings`。

数据分层：

1. 账号索引文件
2. OAuth pending state
3. 平台实例信息
4. 平台附加配置

### 依赖策略

当前 `src-tauri/Cargo.toml` 缺少 `cockpit-tools-main` 平台模块依赖所需的多个 crate。第一阶段允许新增必要依赖，包括但不限于：

- `tokio`
- `url`
- `tiny_http`
- `base64`
- `rand`
- `futures`
- `sysinfo`
- `serde_urlencoded`
- 以及平台功能所需的 Tauri plugins

原则：

1. 仅新增本次迁移确实需要的依赖。
2. 不把 `cockpit-tools-main` 整个 Cargo 配置原样照搬。
3. 优先满足 Linux/Fedora 环境可编译运行。

## 迁移阶段

### 阶段 1

建立统一平台中心骨架并完成基础闭环：

1. 新路由与统一入口
2. API Providers 子页适配
3. Codex/Gemini/Kiro 账号页接入
4. 基础账号操作与 OAuth
5. 平台实例页基础接入

### 阶段 2

补齐平台外围能力：

1. Codex local access
2. Codex wakeup
3. Codex session manager
4. Codex model provider manager
5. Gemini 启动命令/实例注入细节
6. Kiro 注入与额度展示细节

### 阶段 3

进一步与当前产品功能联动：

1. 评估 Codex local access 与 `ModelChatPage` 的桥接
2. 统一部分平台账户事件广播
3. 清理旧模型管理信息架构遗留入口

## 测试策略

必须使用增量 TDD：

1. 先补路由与页面切换测试
2. 再补 store 与 service 接口测试
3. 再补平台页渲染与关键交互测试
4. 最后补 Tauri 命令层验证

优先覆盖：

1. 旧 `model-providers` 路由兼容
2. `ModelChatPage` 不回归
3. OAuth 流程状态切换
4. 账号导入导出
5. 平台子页切换与懒加载

## 风险

1. `cockpit-tools-main` 后端依赖面较大，直接整块迁入会拖高编译风险。
2. 当前项目 `main.rs` 体积已经很大，若不拆模块会显著加剧维护问题。
3. 平台页原有组件数量多，若全部首轮同步加载，会重新引入设置页卡顿问题。
4. 现有项目样式体系与 `cockpit-tools-main` 存在差异，需要做 UI 收口。

## 决策

1. 采用统一平台中心方案，而不是双轨并行的“旧 provider 页 + 新账号中心”。
2. 平台账户与 API Provider 分层，不混写到 `AppSettings`。
3. Kiro 第一阶段完整接入账号中心/实例/注入，但不接入 `ModelChatPage`。
4. 前端优先复用成熟模块，后端采用模块化迁入，不做整仓硬拷贝。
