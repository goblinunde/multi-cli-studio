# Kiro CLI Minimal Integration

## Goal

Add `kiro` as a first-class usable CLI inside Multi CLI Studio's main workspace flow.

## Scope

- Add `kiro` to shared frontend agent types and UI selectors.
- Detect `kiro-cli` in desktop runtime checks.
- Allow chat/task execution through host-managed shell dispatch.
- Allow automation execution mode to target `kiro`.
- Keep Kiro account-center support unchanged.

## Non-goals

- No Kiro-specific session resume implementation in this pass.
- No Kiro-specific model picker implementation in this pass.
- No deep Kiro resource introspection beyond basic runtime presence.

## Runtime assumptions

- Use `kiro-cli` instead of `kiro` to avoid IDE/router ambiguity.
- Use `kiro-cli -V` for version detection.
- Use `kiro-cli chat --no-interactive` for headless execution.
- Use `--trust-all-tools` for write-capable turns.
- Use `--trust-tools=read,grep` for read-only / plan-oriented turns.

These assumptions are based on Kiro's current CLI docs.

## Integration points

1. Frontend shared types and defaults
2. CLI selector and agent tabs
3. Browser fallback mocks
4. Desktop settings/runtime views
5. Rust runtime detection and seeded state
6. Rust chat/task script building
7. Automation execution mode normalization and labels

## Risks

- Some UI surfaces may still assume exactly three CLIs.
- Kiro permission semantics are inferred into the existing generic permission model.
- Kiro model switching remains unsupported until dedicated flags are implemented.
